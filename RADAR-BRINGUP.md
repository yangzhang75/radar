# 真雷达接入手册 (RADAR bring-up)

把真 Navico **HALO** 雷达接进本软件的完整步骤:**物理/网络 → 握手 → 探针验证 → 接 App**。
配套工具 `tools:haloprobe`（纯 JVM 探针）已就绪；LIVE 数据路径 `:comms` / `RadarCommsService` 已就绪。

---

## 0. 原理:HALO 走以太网 UDP 组播

HALO **不是串口**出图，而是在雷达以太网段上发 **UDP 组播**。你的机器（笔记本或 Android 控制台）
必须能收到这些组播——即和雷达在**同一二层网络**，或用 **蒲公英 X5** 把二层桥接过去。

组播端点（已固化在 `comms-core` / `HaloEndpointSet`）：

| 通道 | 地址:端口 | 方向 | 状态 |
|---|---|---|---|
| 图像 A | `236.6.7.8:6678` | 雷达→软件 | ✅ 解析已验证（真数据） |
| 图像 B（双量程） | `236.6.7.13` | 雷达→软件 | ✅ |
| 状态 | `236.6.7.9` | 雷达→软件 | ✅ |
| 目标跟踪数据 | `236.6.7.18:6688` | 雷达→软件 | ⚠ wire 格式待厂商/抓包（我们不依赖它——TT 由回波自提取） |
| 跟踪状态 | `236.6.7.19:6689` | 雷达→软件 | ⚠ 同上 |
| 跟踪控制 | `236.6.7.20:6690` | 软件→雷达 | ⚠ 同上 |
| 握手协商 | 协商通道 01B1/01B2 | 双向 | ✅ 请求/应答已实现 |

> 说明:雷达 TT 通道(18/19/20)的字节格式文档未给全,需真机抓包/SDK 源确认;但本软件的雷达目标是
> **从图像通道(236.6.7.8)自己做点迹提取+跟踪**长出来的,所以即使 TT 通道未解,防撞功能照常工作。

## 1. 网络接入(二选一)

**A. 现场同网段**
1. 雷达以太网口 → 交换机 → 你的机器（或经 Navico 接口盒）。
2. 机器网卡设为雷达网段同一子网（HALO 常见 `10.x` / `192.168.x`；以雷达实际为准）。
3. 交换机需允许**组播/IGMP** 透传（多数默认可；托管交换机确认 IGMP snooping 不误丢）。

**B. 远程经蒲公英 X5**
1. 蒲公英一端接雷达网段，另一端在你机器上装客户端，组成**二层桥接**网络。
2. 确认蒲公英虚拟网卡 **支持组播**；若组播不透（蒲公英偶发），用探针的 `--manual-ip` 走单播回退。

验证网络通不通(能 ping 到雷达 IP 即二层已通):
```
ping <雷达IP>
```

## 2. 用探针验证(先别碰 App —— 最快最稳)

```
# 构建
./gradlew :tools:haloprobe:installDist
BIN=./tools/haloprobe/build/install/haloprobe/bin/haloprobe

# 在线抓取(自动握手 + 每 8s 看门狗防待机 + 打印各通道收包/解码 + 存抓包)
$BIN --iface=<蒲公英/雷达网卡名> --record=cap.bin --duration=60

# 组播不透时,按手动 IP 单播回退
$BIN --iface=<网卡> --manual-ip=<雷达IP> --record=cap.bin
```

参数:
- `--iface=<name>` 指定走哪块网卡(蒲公英虚拟网卡名);不填自动挑一块 up/非回环/支持组播的。
- `--manual-ip=<ip>` 跳过组播握手,按手动 IP 走(组播不通时用)。
- `--no-handshake` 只被动监听,不发 01B1。
- `--record=<file>` 把每个数据报存成抓包(之后可离线回放/逐字段核对)。
- `--duration=<秒>` 抓取时长(默认 60)。

看什么(判断成功):
- `握手: 成功 ✓ 雷达=<ip> serial=<...>` → 双向链路建立。
- `IMAGE <n> | STATUS <n> | TARGET <n>` 每通道收包数在涨 → 数据在来。
- 若 `一个包都没收到` → 见第 5 节排查。

## 3. 离线验证(把 `cap.bin` 发给我，或本地跑)

```
# 解码 + 跑跟踪管线 step1(点迹提取);多圈数据可验证完整跟踪
$BIN --decode-file=cap.bin

# 若"满量程"数值不合理(过小/过大),复核量程单位(见第 4 节)
$BIN --decode-file=cap.bin --range-unit=dm
```
输出会给:辐条数、方位跨度、**提取出的点迹(距离/方位/峰值)**,以及量程单位是否可疑的告警。

---

## 4. 确认量程单位(一次性标定)

真抓包发现:`rangeCellSize` 字段按**毫米**解得的满量程过小(疑似单位是**分米 dm**)。
用 `--range-unit=mm|dm|cm` 在离线解码里对比,选出让满量程合理(与雷达当前量程档吻合)的那个。
确认后我把 LIVE 路径 `CommsConfig.rangeUnitToMm` 定死(现默认 mm，可传 `SpokeParser.RANGE_UNIT_DM`)。
> 这是记录在案的协议歧义项,需厂商或真机标定确认。

---

## 5. 接进 App(LIVE 模式)

1. Android 控制台与雷达同网段 / 蒲公英。
2. App 顶部把数据源从 **SIMULATION 切到 LIVE** → 绑定前台 `RadarCommsService`
   (它做同样的握手 + 组播摄取,跑在专用实时调度上 + 看门狗)。
3. LIVE 下:回波实时渲染、目标由回波自提取跟踪、CPA/TCPA/警戒圈/新丢/容量报警、conning 全部生效。

---

## 6. 排查(收不到包)

| 症状 | 排查 |
|---|---|
| 一个包都没收到 | ① 机器是否在雷达网段(能 ping 雷达 IP);② `--iface` 选对网卡了吗(尤其蒲公英虚拟网卡);③ 防火墙放行 UDP 6656-6690 |
| ping 通但无组播 | 交换机 IGMP snooping / 蒲公英组播不透 → 用 `--manual-ip` 单播回退 |
| 握手超时(收不到 01B2) | 雷达在待机?先加 `--manual-ip` 被动收流验证物理通路;确认雷达处于发射态 |
| 收到图像但解不出辐条 | 极少见(解析已对真数据验证);把 `cap.bin` 发我用 `--decode-file` 逐帧核对 |
| 图像正常但目标不出 | 目标需数圈稳定跟踪确认(A.823:1 分钟内出稳定航迹);多等几圈 |

## 附:最省事的启动闭环

不必先搞定 Android 控制台。找台带 JDK 的笔记本接雷达网段(或蒲公英):
```
./gradlew :tools:haloprobe:installDist
./tools/haloprobe/build/install/haloprobe/bin/haloprobe --iface=<网卡> --record=cap.bin --duration=60
```
把 `cap.bin` 发回 → 即可验证跟踪链 + 标定量程单位,再推进 App / 认证。
